## Creating tasks

You don't need a special UI. Just describe what you want in a conversation:

```
"Suggest AI tech short articles everyday so I can post on sub"
```

NIOM's [Skill Tree](/guides/skill-tree) detects the scheduling intent and automatically creates a background task. You'll see:
1. A task created with a goal, schedule, and execution plan
2. The first run starts immediately
3. Progress streams live in the task panel

You can also create tasks via the API:
```bash
curl -X POST http://localhost:3001/tasks \
  -H "Content-Type: application/json" \
  -d '{
    "goal": "Research the latest AI news and write a brief summary",
    "taskType": "recurring",
    "schedule": { "interval": "daily" }
  }'
```

## Live streaming

Tasks stream their progress via SSE — the same way conversations do. You see every tool call as it happens:

```
⏳ Run #1 started
  🔍 Searched: "latest AI news March 2026"
  🌐 Fetched: techcrunch.com/ai-roundup
  🌐 Fetched: arxiv.org/latest/cs.AI
  📝 Wrote: ai_research_summary.md
✅ Run #1 completed (45.2s, 7 tools)
```

The task panel auto-expands running tasks and shows:
- Live tool status ("Running: Deep Research")
- Each tool call with its input and output
- Phase progress for multi-step tasks
- Quality score on completion

**No polling.** The frontend builds its state directly from SSE events. When a tool call finishes on the backend, it appears in the UI immediately.

## Steering

Steering replaces the old approval model. Instead of approve/deny, you **guide** running tasks with comments:

```
"Focus more on open-source models in the next run"
```

Post a steering comment from the task panel. It flows into the next run as additional context. The agent reads your feedback and adjusts its approach.

Steering works because:
- Comments are stored in task memory
- Each run receives the full history of previous steering comments
- The agent adapts based on accumulated feedback

## Task memory

Every task maintains a persistent memory that includes:
- **Previous run outputs** — what was produced, what worked
- **Steering comments** — your accumulated feedback
- **Tool usage history** — which tools succeeded, which failed
- **Quality evaluations** — how each run was scored

This means run #5 is better than run #1 — the agent learns from its own execution history.

## Task lifecycle

Tasks are persistent — they survive overlay closes, sidecar restarts, and machine reboots:

```
draft → planned → scheduled → flowing (executing) → completed
                      ↑             ↓
                   resume         pause
                      ↑             ↓
                    paused ←────────┘
```

**Actions you can take:**
- **Run Now** — trigger an immediate execution, regardless of schedule
- **Pause / Resume** — temporarily stop or restart a task
- **Steer** — post a comment that influences the next run
- **Edit** — change the goal, schedule, or other settings
- **Delete** — removes the task and cleans up non-workspace artifacts

## Artifacts from tasks

Files created during task execution are automatically registered as [artifacts](/guides/artifacts):
- Find them in the task's detail panel
- Or navigate to `~/.niom/artifacts/{task-id}/`
- If the task has a workspace, artifacts go to your project instead

## Native notifications

Task events trigger real desktop notifications (via Tauri — not just in-app alerts):
- ✅ **Completed** — "Task Complete: Research AI news..."
- ❌ **Failed** — with error description
- These notifications work even when the overlay is closed

## SSE event types

For developers building integrations, the task event stream (`GET /tasks/events`) emits:

| Event | Payload | When |
|:------|:--------|:-----|
| `task:start` | `{ taskId, runId, runNumber, startedAt, phases }` | A run begins |
| `task:tool` | `{ taskId, runId, tool, input, output }` | A tool call completes |
| `task:phase` | `{ taskId, phase, status }` | A phase changes status |
| `task:complete` | `{ taskId, runId, status, output, durationMs, qualityScore }` | A run finishes |
| `task:error` | `{ taskId, runId, error }` | A run fails |
| `task:steer` | `{ taskId, comment }` | A steering comment is posted |
