When you ask NIOM to do something, it doesn't throw every tool at the problem. It routes your request through a **Skill Tree** ‚Äî a graph of tools organized by domain ‚Äî and selects the 4-8 tools most likely to succeed.

This is what makes NIOM faster, cheaper, and more accurate than agents that load all tools for every request.

## How it works

```
Your message
  ‚Üí Skill Tree analyzes intent
    ‚Üí Routes to a domain (Code, Research, Creative, etc.)
      ‚Üí Selects focused tool subset (4-8 tools)
        ‚Üí Agent executes with just those tools
```

<Steps>
<Step title="Classify domain">
  A fast model (~200ms) reads your message and maps it to one of NIOM's built-in skill domains:

  | Domain | Example requests |
  |:-------|:----------------|
  | **Code** | "Refactor this module", "Find bugs in my project" |
  | **Research** | "Research quantum computing", "Summarize this paper" |
  | **Creative** | "Write a blog post about AI", "Draft an email" |
  | **Business** | "Analyze this spreadsheet", "Create a project plan" |
  | **Personal** | "Set a reminder", "Organize my files" |
  | **General** | "What's the weather?", "Tell me a joke" |
</Step>
<Step title="Build a Skill Path">
  Based on the domain, NIOM constructs a **Skill Path** ‚Äî a ranked list of tools weighted by:
  - **Domain relevance** ‚Äî which tools belong to this skill domain
  - **Co-occurrence data** ‚Äî which tools tend to succeed together (learned from your usage)
  - **Query embedding similarity** ‚Äî how close your request is to each tool's capabilities

  The result is a focused subset of 4-8 tools instead of the full 15+.
</Step>
<Step title="Detect execution mode">
  Before executing, a lightweight scanner checks for scheduling intent:
  - "do this **every morning**" ‚Üí creates a recurring background task
  - "**research deeply**" ‚Üí routes to deep research mode
  - "**monitor** my folder" ‚Üí creates a one-shot observation task

  This happens automatically ‚Äî you don't need to manually choose "background task" vs "direct response."
</Step>
<Step title="Execute with the right tools">
  The agent runs with only the selected tools. Fewer tools means:
  - **Faster** ‚Äî less context for the LLM to process
  - **Cheaper** ‚Äî smaller prompts = lower token cost
  - **More accurate** ‚Äî the model isn't confused by irrelevant tools
</Step>
</Steps>

## Built-in skill domains

NIOM ships with 6 domain packs. Each pack defines which tools are available:

  <Accordion title="üîç Research">

  <Accordion title="‚úèÔ∏è Creative">

  </Accordion>
  <Accordion title="üìä Business">

## The co-occurrence graph

As you use NIOM, it tracks which tools succeed together. If `webSearch` ‚Üí `fetchUrl` ‚Üí `writeFile` frequently co-occur in your research tasks, the Skill Tree learns this pattern and pre-selects all three when you ask for research.

This data is stored locally at `~/.niom/skill-graph.json`. It's not sent anywhere ‚Äî your usage patterns stay on your machine.

Over time, tool selection becomes personalized to your workflow.

## Schedule detection

NIOM's Skill Tree includes a 2-tier schedule detection system:

**Tier 1 ‚Äî Pattern matching** (zero latency):
- Detects obvious patterns: "every day", "weekly", "in 2 hours"
- Triggers immediately ‚Äî no LLM call needed

**Tier 2 ‚Äî LLM validation** (conditional):
- Fires only when the query is ambiguous or combines temporal language with actions
- Uses the fast model to confirm whether scheduling is intended
- Only applies if confidence > 0.7 ‚Äî prevents false positives

If scheduling intent is detected, the request is automatically routed to the background task system instead of trying to execute everything in one shot.

## Seeing it in action

You don't need to think about the Skill Tree ‚Äî it works silently. But if you're curious, you can see it:

1. **Console logs** ‚Äî the sidecar logs which domain was selected, which tools were loaded, and why
2. **Skill graph file** ‚Äî check `~/.niom/skill-graph.json` to see co-occurrence data
3. **Task creation** ‚Äî when the Skill Tree detects scheduling intent, it creates a task you can see in the Tasks panel

<Tip>
  The Skill Tree is designed to be invisible when it works well. The best indicator is that NIOM consistently picks the right tools without being told which ones to use.
</Tip>
