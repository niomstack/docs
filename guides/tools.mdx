NIOM ships with a set of powerful built-in tools. The [Skill Tree](/guides/skill-routing) automatically picks the right ones for each request — you never need to specify which tool to use.

## Tool inventory

### File system

| Tool | What it does | Needs approval? |
|:-----|:------------|:----------------|
| **read** | Reads file contents or lists directory contents. Auto-detects file vs directory. | ❌ No |
| **write** | Creates or modifies files. Shows a diff preview before writing. | ✅ Yes |
| **listDirectory** | Lists a directory tree with file sizes and types. | ❌ No |

<Tip>
  The `read` tool is smart — give it a file path and it returns the contents. Give it a directory and it returns a listing. No need to think about which tool to use.
</Tip>

### Shell

| Tool | What it does | Needs approval? |
|:-----|:------------|:----------------|
| **runCommand** | Executes shell commands (`ls`, `git`, `npm`, etc.) | ✅ Always |

Shell commands always require your approval — even in trusted mode. NIOM shows you the exact command before running it.

### Web

| Tool | What it does | Needs approval? |
|:-----|:------------|:----------------|
| **webSearch** | Searches the web using DuckDuckGo | ❌ No |
| **crawl** | Extracts readable content from a URL | ❌ No |
| **fetchUrl** | Raw HTTP fetch for APIs and URLs | ❌ No |

### System

| Tool | What it does | Needs approval? |
|:-----|:------------|:----------------|
| **systemInfo** | Returns OS, CPU, memory, and environment details | ❌ No |

### Artifacts

| Tool | What it does | Needs approval? |
|:-----|:------------|:----------------|
| **proposeArtifact** | Creates a file with a diff preview for your review | ✅ Preview |

## Tool approval

NIOM uses a **trust-based approval system**. When a tool needs approval:

1. A **tool approval card** appears in the chat showing what the tool wants to do
2. You click **Approve** or **Reject**
3. If approved, the tool executes and the result appears below

### Session trust

When you approve a tool for a specific path, NIOM remembers that trust for the rest of the session:

- Approving `write` to `/Users/you/project/src/` trusts that tool for that path
- Trust is **path-scoped** — approving one directory doesn't trust all directories
- Trust is **session-scoped** — clears when you restart NIOM (30-minute TTL)
- `runCommand` is **never auto-trusted** — every command requires explicit approval

<Note>
  This is a safety net, not a hindrance. After a few approvals in the same project, NIOM flows smoothly because it remembers what you've trusted.
</Note>

## Self-correction

NIOM's agent doesn't just call tools — it **interprets results and self-corrects**. Every tool returns a structured result:

```
success → Continue with the data
partial → Some results, try refining
error   → Read the suggestions, retry with different args
```

For example, if `read("/src/index.ts")` returns "file not found", the agent might:
1. Try `read("/src/")` to list the directory
2. Discover the file is `index.tsx` (not `.ts`)
3. Read the correct file automatically

This happens without your intervention — up to 3 self-correction attempts per tool call.

## Adding more tools via MCP

NIOM supports the **Model Context Protocol (MCP)**, which connects it to 13,000+ external tools:

| MCP Server | What it adds |
|:-----------|:-------------|
| GitHub | Repositories, issues, PRs, commits |
| Slack | Messages, channels, search |
| Google Drive | Documents, sheets, files |
| Postgres | Database queries |
| Gmail | Email reading and composition |

MCP support is a planned integration. The tool infrastructure is ready, and MCP servers will be configurable through the Settings UI.

<Tip>
  NIOM's Skill Tree automatically incorporates MCP tools into its routing graph. External tools get the same intelligent routing as built-in tools.
</Tip>
