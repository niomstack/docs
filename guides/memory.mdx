NIOM's memory system is what separates it from every other AI chat app. While others start from zero every conversation, NIOM **remembers**.

## How it works

After every conversation, NIOM runs a lightweight extraction pass that identifies:

- **Facts** — "I work at Acme Corp", "My project uses Next.js 15"
- **Preferences** — "I prefer TypeScript strict mode", "I like dark backgrounds"
- **Patterns** — "When I ask about code, I usually mean the niom project"
- **Corrections** — "Actually, I meant Python, not JavaScript" updates the existing memory

These are automatically organized into a **NIOM Context Filesystem (NCF)** — a tiered hierarchy that lives at `~/.niom/context/`.

<Note>
  Memory extraction happens automatically. You don't need to do anything special — just talk to NIOM naturally.
</Note>

## The three tiers

NIOM uses an OpenViking-inspired tiered loading system to optimize token usage:

| Tier | Size | What it contains | When it's used |
|:-----|:-----|:-----------------|:---------------|
| **L0 (Abstract)** | ~100 tokens | One-line summary of each memory | Always loaded — every conversation |
| **L1 (Overview)** | ~2K tokens | Structured detail, key facts | Loaded when the topic is relevant |
| **L2 (Full Detail)** | Variable | Complete context, full histories | Loaded on explicit demand |

This means NIOM always has broad awareness of everything it knows about you (L0), loads relevant detail when needed (L1), and pulls full context only when deep-diving (L2).

## Recall mode

The **Recall** toggle in the prompt box controls whether NIOM injects memories into the current conversation.

- **Recall ON** — NIOM searches its memory for relevant context and includes it in the system prompt. The model sees what NIOM knows about you.
- **Recall OFF** — Clean conversation with no memory injection. Useful for when you want a generic response.

<Tip>
  Recall is ON by default. Toggle it off when you want NIOM to respond without any personal context — like when helping someone else.
</Tip>

## What NIOM remembers

The memory filesystem is organized by category:

```
~/.niom/context/
├── user/                    # About you
│   ├── brain.json           # Core identity: name, timezone, preferences
│   └── preferences/         # Domain-specific preferences
│       ├── coding.json      # "Uses TypeScript, prefers strict mode"
│       └── communication.json
├── projects/                # Your workspaces
│   └── <workspace-hash>/
│       └── context.json     # Tech stack, conventions, structure
└── agent/                   # Operational context
    └── memories/            # Extracted facts and patterns
```

## Managing your memories

Open **Settings** → **Context** tab to:

- **View all memories** — See everything NIOM has learned, organized by category
- **Delete individual memories** — Remove anything you don't want NIOM to remember
- **View the Context Graph** — A visual React Flow graph showing the relationships between your memories, projects, and preferences
- **Check stats** — Total memory count, project count, index size

<Note>
  NIOM is transparent. You can see, edit, and delete everything it remembers. There are no hidden memories or data NIOM keeps that you can't see.
</Note>

## Project awareness

NIOM automatically scans your workspace directories and indexes:

- **Tech stacks** — frameworks, languages, package managers detected from config files
- **Conventions** — code style, directory structure patterns
- **Project metadata** — name, root path, when it was last analyzed

This context is injected when you're working within a recognized project, so NIOM knows you're in a "React + TypeScript + Vite project" before you tell it.

## How memory makes NIOM better

Here's a concrete example of memory in action:

**Day 1:**
> You: "Help me set up a new Node.js project"
> NIOM: Sets up a basic Node.js project with JavaScript

**Day 2** (after NIOM has learned your preferences):
> You: "Help me set up a new Node.js project"
> NIOM: Sets up a TypeScript project with strict mode, ESLint, your preferred structure — because it remembers your preferences from previous conversations

**This is the "gets smarter with use" property.** Every conversation teaches NIOM something. Over time, it stops asking clarifying questions it already knows the answer to.
