NIOM has an unusual architecture: a **hybrid daemon** that splits native OS access from AI intelligence into two cooperating processes. This isn't a wrapper around an LLM — it's an OS-level intelligence layer.

## The two processes

```
┌──────────────────────────────────────────────────────────────────────┐
│                           NIOM Daemon                                │
│                                                                      │
│  ┌─── Rust (Tauri) ──────────┐  ┌──── Node.js Sidecar ─────────────┐│
│  │                           │  │                                   ││
│  │  System tray icon         │  │  🌳 Skill Tree (domain routing)  ││
│  │  Global hotkeys           │  │  🧠 Agent engine (4-tier)        ││
│  │  Overlay window           │  │  ⚡ Background tasks (SSE)       ││
│  │  Native notifications     │  │  🤲 Built-in tools (15+)        ││
│  │  Sidecar lifecycle        │  │  🔍 Search (DDG/Tavily/Brave)   ││
│  │                           │  │  🌐 MCP client (13,000+ tools)  ││
│  │  That's it.               │  │  🖥️ Computer use                ││
│  │  ~200 lines of Rust.      │  │  💬 SSE streaming               ││
│  │                           │  │  📎 Artifacts system             ││
│  │                           │  │  🧩 Long-term memory (brain)    ││
│  │                           │  │  🩺 Self-healing                ││
│  └───────────────────────────┘  └───────────────────────────────────┘│
│                                                                      │
│  ┌──── Frontend (React + Vite + Tailwind v4) ───────────────────────┐│
│  │  Tauri IPC → Rust (OS-native only)                               ││
│  │  HTTP/SSE  → Node.js (everything else)                           ││
│  └──────────────────────────────────────────────────────────────────┘│
└──────────────────────────────────────────────────────────────────────┘
```

## Why split it this way?

An earlier version of NIOM kept everything in Rust. It was 2,800 lines across 5 crates — and couldn't match what Node.js does in 120 lines:

| Task | Rust (before) | Node.js (now) |
|:-----|:---:|:---:|
| File read/write | 60 lines | 2 lines |
| Shell execution | 40 lines | 3 lines |
| Web search | 105 lines (and broken) | 3 lines |
| Intent classification | 440 lines | 15 lines |
| Agent loop | 891 lines | 30 lines |
| Model manager | 597 lines | 1 line |
| **Total** | **~2,800 lines** | **~120 lines** |

The TypeScript AI ecosystem (Vercel AI SDK, MCP SDK, etc.) is **10-100x more mature** than Rust for this. Moving intelligence to Node.js and keeping only OS-native operations in Rust gives the best of both worlds.

## System layers

NIOM's intelligence operates through multiple interconnected layers:

### Layer 1: Skill Tree (routing)

Every request enters through the [Skill Tree](/architecture/skill-tree) — a graph of tools organized by domain. It classifies your intent, selects the right 4-8 tools (from 15+ available), and determines execution mode (direct, background task, or scheduled).

```
User message → Domain classification → Skill Pack → Tool subset → Execution mode
```

The Skill Tree is what makes NIOM different from a generic agent. Instead of dumping all tools into the prompt, it routes intelligently. See the [Skill Tree architecture](/architecture/skill-tree) deep-dive.

### Layer 2: Agent engine (execution)

The agent pipeline takes the Skill Tree's output and executes with adaptive depth:

```
Analyze → Route by complexity → Execute with tools → Evaluate quality → Respond (or refine)
```

For complex tasks, evaluation creates a refinement loop (up to 3 rounds). For simple queries, the pipeline is nearly instant. See the [Agent Engine guide](/guides/agent-engine).

### Layer 3: Artifacts + memory (persistence)

Everything the agent produces is tracked:
- **Artifacts** — files created during execution are auto-registered and linked to conversations/tasks. See [Artifacts](/guides/artifacts).
- **Brain memory** — facts, preferences, and patterns extracted from conversations. Injected into every future interaction. See [Memory](/guides/memory).
- **Task memory** — per-task execution history, steering comments, and tool results. Makes each run smarter than the last.

### Layer 4: SSE event system (streaming)

Both conversations and tasks stream their progress via Server-Sent Events:
- **Conversations** — AI SDK UIMessageStream protocol (text deltas, tool calls, reasoning)
- **Tasks** — custom event types: `task:start`, `task:tool`, `task:phase`, `task:complete`, `task:error`

The frontend builds its state directly from SSE events — no polling. See [Task Events API](/api-reference/tasks/events).

## Data flow

Here's what happens when you type something in the overlay:

```
You type in the overlay
  → HTTP POST /run → Node.js sidecar
    → Skill Tree routes to domain + selects tools (~200ms)
    → Agent executes with selected tools (capable model)
    → Artifacts auto-registered if files created
    → Brain extracts facts asynchronously after save
    → SSE streams response back to the overlay
```

OS-level operations (hotkeys, tray, notifications) go through **Tauri IPC** → Rust. Everything else goes through **HTTP** → Node.js. Clean separation.

## Privacy

<Note>
  **Local-first is non-negotiable.** All data lives on your machine at `~/.niom/`. Zero telemetry. Zero analytics. The only network calls are to your chosen LLM provider — and even that's optional (Ollama works fully offline).
</Note>
