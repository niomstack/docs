The Node.js sidecar (`niom-ai/`) is where all intelligence lives. It's a standalone HTTP server that the Rust shell spawns and manages as a child process.

## Key dependencies

| Package | What it does |
|:--------|:------------|
| `ai` (v6) | Vercel AI SDK — `streamText()`, `generateText()`, `Output.object()`, tool calling |
| `@ai-sdk/anthropic` | Direct Anthropic provider (no gateway needed) |
| `@ai-sdk/openai` | Direct OpenAI provider |
| `@ai-sdk/google` | Direct Google provider |
| `hono` | HTTP framework (lightweight, fast, Express-like) |
| `@modelcontextprotocol/sdk` | MCP client for connecting to external tool servers |
| `zod` | Schema validation for structured output (intent analysis, schedule detection) |
| `@mozilla/readability` + `jsdom` | Web page content extraction (for `fetchUrl`) |

## Boot sequence

Here's what happens when NIOM starts:

1. Tauri spawns the sidecar as a child process
2. Sidecar loads config from `~/.niom/config.json`
3. Initializes AI providers (direct — no gateway), MemoryStore, TaskManager, ArtifactManager
4. Loads Skill Packs from `skills/packs/` → builds SkillRegistry
5. Loads co-occurrence graph from `~/.niom/skill-graph.json`
6. Connects to configured MCP servers
7. Starts the Hono HTTP server on `localhost:3001`
8. Returns `200 OK` on `GET /health`
9. Tauri polls health every 5 seconds — auto-restarts on crash (max 3 attempts)

## Streaming protocols

The sidecar uses two SSE protocols:

### Conversation streaming (AI SDK UIMessageStream)

| Event | When it fires |
|:------|:-------------|
| `reasoning-start/delta/end` | During the analysis phase |
| `text-delta` | As the response streams to the overlay |
| `tool-input-start/available` | When a tool call begins and its arguments are ready |
| `tool-output-available` | When a tool returns its result |
| `tool-approval-request` | When a destructive action needs your OK |
| `step-boundary` | Between steps in multi-step execution |
| `finish` | Stream complete |

### Task streaming (custom events)

| Event | Payload | When it fires |
|:------|:--------|:-------------|
| `task:start` | `{ taskId, runId, runNumber, startedAt, phases }` | A run begins |
| `task:tool` | `{ taskId, runId, tool, input, output }` | A tool call completes |
| `task:phase` | `{ taskId, phase, status }` | A phase changes status |
| `task:complete` | `{ taskId, runId, status, output, durationMs, qualityScore }` | A run finishes |
| `task:error` | `{ taskId, runId, error }` | A run fails |

<Tip>
  If you're building on top of NIOM's API, use any AI SDK-compatible SSE client for conversations. For task events, use a standard `EventSource`. See the [Task Events API](/api-reference/tasks/events).
</Tip>
